# GENERATIVE PUZZLE - v2.0 平台化重构 PRD
## Product Requirements Document
**版本 v2.0 · 2026-05 · 基准代码 generative-puzzle v1.4.19**

---

# 0. 文档定位

v2.0 不是在现有拼图页上追加几个开关，而是把 Generative Puzzle 从“单局生成式拼图游戏”升级为“可持续运营的生成式拼图平台”。

平台化的核心目标是：

1. 将现有拼图引擎拆分为可复用的游戏内核、模式壳、数据管线和运营能力。
2. 用四个模式承接不同玩家意图：探索、竞速、硬核挑战、放松沉浸。
3. 用确定性生成、独立榜单、个人档案和渐进式解锁构建长期留存。
4. 以 2026 年 AI 时代玩家需求为准绳：新鲜但不廉价，个性化但不操控，公平但不压迫，生成式但可信。

## 0.1 v2.0 相比 v1.1 的关键修订

| 项目 | v1.1 方案 | v2.0 修订 |
| --- | --- | --- |
| 产品定位 | 四模式架构设计 | 平台化重构 PRD，四模式是平台首批产品形态 |
| 首发范围 | P0 同时承担路由、每日挑战、独立榜单、日期种子 | 拆为 Foundation、Launch、Expansion 三层，先打地基再上完整玩法 |
| 准入机制 | 每日挑战作为硬钥匙，禅意 7 天后解锁 | 改为软仪式：自由和禅意低摩擦开放，深层资产和特权通过连续打卡解锁 |
| 每日挑战一致性 | 100% 像素级一致 | 改为逻辑确定性一致：形状拓扑、切割、颜色、初始旋转、逻辑散开坐标一致；渲染允许设备自适应 |
| AI 时代表达 | 隐含在“生成式”里 | 明确采用“AI/生成式能力沉到底层”的策略，不以可见 AI 内容作为卖点 |
| 数据隔离 | 单表加 `game_mode` 和 `challenge_date` | 定义会话表、排行榜投影表、盲拼档案、禅意时长、每日状态的分层数据模型 |
| 里程碑 | P0-P5 功能列表 | R0-R4 平台发布列车，每阶段有验收标准和风险边界 |

# 1. 战略判断

## 1.1 项目发展判断

当前项目已经具备 v2.0 平台化的基础：

- 已有可玩的 Canvas 拼图体验。
- 已有三种切割类型、三种形状类型、难度体系、旋转和提示系统。
- 已有计分器、排行榜、Supabase 云记录和后台看板雏形。
- 已有移动端适配、音效、震动、国际化和无框 UI 经验。
- 已有 `@generative-puzzle/game-core` 包，适合继续沉淀平台内核。

但当前项目仍是单入口单模式形态：

- 首页直接挂载游戏界面，而不是模式大厅。
- 全局状态未显式携带 `gameMode`。
- 生成链路仍依赖不可设种的随机数。
- Supabase 排行榜仍以单一 `difficulty` 聚合，无法承接模式隔离。

因此 v2.0 的正确方向不是“先做四个大功能”，而是先建立平台边界：模式、路由、种子、数据、结算、运营。

## 1.2 2026 年 AI 时代玩家需求判断

2026 年玩家对 AI 游戏的真实需求不是“看到 AI 在生成”，而是：

| 玩家需求 | 对 Generative Puzzle 的含义 |
| --- | --- |
| 无限新鲜感 | 每局都不同，但规则和手感稳定 |
| 公平可信 | 每日挑战必须同题同参，结果可验证 |
| 低摩擦进入 | Web 首访不能被复杂注册、硬锁和强制任务挡住 |
| 个性化节奏 | 玩家可以选择竞争、挑战或放松，而不是被单一分数系统绑架 |
| 可炫耀表达 | 排名、盲拼评级、截图、档案称号要能表达身份 |
| 情绪友好 | 卡关后要能尊严离场，疲惫时要能无压力游玩 |
| 反 AI 廉价感 | 不用“AI 生成内容”当噱头，避免低质、失控和版权焦虑 |

v2.0 的产品策略是：**生成式能力沉入规则层，玩家感知到的是新鲜、公平、可复玩，而不是被迫消费 AI 标签。**

# 2. 产品愿景

## 2.1 一句话愿景

Generative Puzzle v2.0 是一个每天都有新题、每局都有新形状、每个玩家都能找到自己节奏的生成式拼图平台。

## 2.2 设计原则

### 原则一：模式服务心理状态

模式不是难度菜单，也不是功能开关。每个模式对应一种玩家意图：

- 自由模式：我想掌控和探索。
- 每日挑战：我想和世界在同一条件下比一次。
- 盲拼模式：我想证明我的空间推理能力。
- 禅意模式：我想放松，不想被评价。

### 原则二：公平必须可验证

每日挑战和排行榜必须基于确定性生成、服务端参数校验和模式隔离。玩家可以输，但不能怀疑题目不一样。

### 原则三：生成式不等于失控

生成系统必须有边界：

- 形状可变，但可拼性稳定。
- 难度可变，但参数透明。
- 每日题可运营调控，但不能暗改已发布题目。
- 个性化可以推荐，但不能操控玩家付出。

### 原则四：放松模式不应被锁在留存任务之后

禅意模式承担情绪出口和传播价值。v2.0 不再把它作为 7 日连续打卡后的硬功能奖励，而是将高级环境音、主题和称号作为长期奖励。

### 原则五：平台边界优先于单点功能

v2.0 所有功能必须围绕五个边界建设：

1. 路由边界。
2. UI Shell 边界。
3. 游戏状态边界。
4. 数据存储边界。
5. 运营看板边界。

# 3. 目标与非目标

## 3.1 v2.0 目标

| 目标 | 说明 |
| --- | --- |
| 平台化入口 | 首页从直接进入游戏改为 Lobby 模式大厅 |
| 模式独立 | 四个模式拥有独立路由、UI Shell、结算、数据归属 |
| 确定性生成 | 引入 seedable PRNG，支撑每日同题和可复盘 |
| 数据升级 | Supabase 支持 `game_mode`、`challenge_date`、attempt、blind grade、zen duration |
| 每日挑战 MVP | 全服同题、首完成、一次重拼、独立日榜 |
| 禅意模式首发 | 低摩擦开放，无计分、无排行榜、沉浸体验 |
| 盲拼模式扩展 | 隐藏轮廓、90 度旋转、评级、降级完成 |
| 拼图师档案 | 将玩家行为转化为身份表达和长期进度 |

## 3.2 v2.0 非目标

| 非目标 | 原因 |
| --- | --- |
| 不做聊天式 AI NPC | 与拼图核心玩法弱相关，增加成本和风险 |
| 不做开放文本生成图片 | 版权、质量、审核和成本不可控 |
| 不做强 PVP 匹配 | 当前体量更适合异步竞速和个人档案 |
| 不做付费系统 | v2.0 先验证留存与模式结构 |
| 不做复杂社交关系链 | 先用排行榜、分享图和档案承接轻社交 |

# 4. 用户分层

## 4.1 核心用户类型

| 用户类型 | 行为特征 | 核心需求 | 推荐入口 |
| --- | --- | --- | --- |
| 新访客 | 第一次打开，未理解规则 | 立刻玩到，不被注册挡住 | 自由模式、禅意模式 |
| 休闲玩家 | 偶尔玩，喜欢低压力 | 放松、好看、短会话 | 禅意模式、自由模式 |
| 日活玩家 | 每天回来一次 | 同题竞争、连续记录 | 每日挑战 |
| 高手玩家 | 追求技巧表达 | 高难度、评级、证明自己 | 盲拼模式 |
| 回流玩家 | 过去玩过，隔几天回来 | 继续记录、快速恢复 | Lobby 最近配置、每日挑战 |

## 4.2 准入策略

v2.0 采用“低摩擦进入 + 软仪式驱动 + 深层奖励解锁”的策略。

| 用户状态 | 可进入模式 | 说明 |
| --- | --- | --- |
| 首访新人 | 自由、禅意、每日挑战 | 不强制登录；每日挑战可试玩但上传正式榜单需登录或虚拟身份 |
| 完成首局后 | 自由、禅意、每日挑战、盲拼入门 | 盲拼建议在完成一局后开放，避免完全无规则认知 |
| 登录玩家 | 全模式 | 开启云记录、日榜、档案、连续打卡 |
| 连续 7 天 | 高级禅意资产、周常边框 | 奖励表达，不锁核心放松模式 |
| 连续 14 天 | 全局仪式豁免、特约拼图师称号 | 不再要求每日挑战作为进入其它模式的钥匙 |

## 5.1 路由设计

| 路由 | 页面 | 说明 |
| --- | --- | --- |
| `/` | Lobby 模式大厅 | 首屏入口，展示四模式状态和最近进度 |
| `/play/free` | 自由模式 | 当前主游戏体验迁移至此 |
| `/play/daily` | 每日挑战 | 受每日状态机保护 |
| `/play/daily/result` | 每日结果回顾 | 已完成或 DNF 后进入 |

| `/play/zen` | 禅意模式 | 极简全屏沉浸 |
| `/profile` | 拼图师档案 | 四维进度和解锁状态 |
| `/scores` | 运营后台 | 从用户总表升级为模式运营看板 |

## 5.2 Lobby 模式大厅

Lobby 是 v2.0 的平台首页，不是营销落地页。

### 模式卡片外显信息

| 模式 | 主信息 | 辅助信息 | CTA |
| --- | --- | --- | --- |
| 自由模式 | 上次配置 | 难度、切割、形状、最佳分 | 继续探索 |
| 每日挑战 | 今日状态 | 倒计时、难度、完成状态、排名 | 开始今日挑战 |
| 盲拼模式 | 当前段位 | S 级覆盖率、最近评级 | 进入盲拼 |
| 禅意模式 | 禅意时长 | 当前环境音、主题、累计时长 | 放松一下 |

### Lobby 设计要求

- 首屏必须直接展示四个模式，不做大篇幅品牌 Hero。
- 卡片是入口，不是说明书；不要用长文教玩家怎么玩。
- 每张卡片展示“现在能做什么”和“我已经做到哪里”。
- 新访客默认高亮自由模式和禅意模式，降低认知压力。
- 日活玩家默认高亮每日挑战。

# 6. 模式架构

## 6.1 ModeController

v2.0 引入 `ModeController` 作为模式分发层。

`ModeController` 不承载拼图算法，只负责：

- 读取当前 `gameMode`。
- 加载模式配置。
- 挂载对应 UI Shell。
- 注入模式规则。
- 分发结算组件。
- 决定记录写入策略。

### 模式枚举

```ts
export type GameMode = 'FREE' | 'DAILY' | 'BLIND' | 'ZEN';
```

### 模式配置

```ts
type ModeConfig = {
  mode: GameMode;
  route: string;
  scoring: 'score' | 'daily_score' | 'blind_grade' | 'none';
  leaderboard: 'free' | 'daily' | 'blind_profile' | 'none';
  showSettings: boolean;
  showScoreHud: boolean;
  showTimer: boolean;
  showHints: boolean;
  showTargetOutline: 'always' | 'memory_then_hidden' | 'never';
  rotationStepDeg: 15 | 90;
  snapThresholdMultiplier: number;
  persistencePolicy: 'best_record' | 'daily_attempt' | 'blind_record' | 'duration_only';
};
```

## 6.2 UI Shell 隔离

| 模式 | UI Shell |
| --- | --- |
| 自由模式 | 全量控制面板、分数、计时、提示、排行榜 |
| 每日挑战 | 无参数面板；显示日期、倒计时、正向计时、重拼标识 |
| 盲拼模式 | 无参数面板；显示正向计时、提示次数、评级目标、降级按钮 |
| 禅意模式 | 全屏无 HUD；只保留极小退出入口和可选音量入口 |

底层拼图组件不得堆砌跨模式 `if-else`。模式差异通过 `ModeConfig` 和注入的规则对象传入。

# 7. 生成系统重构

## 7.1 为什么必须先做

每日挑战、公平排行榜、题目回放、分享复盘都依赖确定性生成。只要核心链路继续直接使用 `Math.random()`，每日同题就无法成立。

## 7.2 PRNG 目标

引入平台级 PRNG：

- 同一 seed + 同一参数生成同一逻辑拼图。
- 不同模块不得自行调用 `Math.random()`。
- 随机源通过 `RandomSource` 注入。
- 生成结果可序列化为 `puzzle_fingerprint`。

### 随机源接口

```ts
type RandomSource = {
  seed: string;
  next(): number;       // [0, 1)
  int(min: number, max: number): number;
  pick<T>(items: T[]): T;
  shuffle<T>(items: T[]): T[];
};
```

## 7.3 确定性一致性定义

v2.0 不承诺所有设备像素级一致，承诺逻辑一致：

| 层级 | 一致性要求 |
| --- | --- |
| 原始形状拓扑 | 必须一致 |
| 切割线/曲线网络 | 必须一致 |
| 碎片数量和顶点顺序 | 必须一致 |
| 颜色分配 | 必须一致 |
| 初始旋转 | 必须一致 |
| 逻辑散开坐标 | 必须一致 |
| Canvas 像素渲染 | 允许因设备尺寸、DPR、字体和抗锯齿差异自适应 |

## 7.4 重构范围

| 模块 | 需求 |
| --- | --- |
| ShapeGenerator | 接收 `RandomSource` |
| OptimizedShapeGenerator | 接收 `RandomSource` |
| cutGenerators | 接收 `RandomSource` |
| NetworkCutter | 接收 `RandomSource` |
| BezierCurve | 接收 `RandomSource` |
| PuzzleGenerator | 接收 `RandomSource`，颜色洗牌确定性 |
| ScatterPuzzle | 接收 `RandomSource`，散开位置和旋转确定性 |
| 调试/加载动画 | 可继续使用非确定性随机，但不得影响游戏结果 |

# 8. 模式边界与策划文档索引

本章只定义 v2.0 平台需要承载的模式边界、路由归属和架构结果，不作为每日挑战、盲拼模式、禅意模式的最终玩法定案。

每日挑战、盲拼模式、禅意模式必须各自维护独立策划文档。只有当对应独立文档完成核心循环、差异化机制、可玩性验证、UI 流程、数据规则和验收标准后，才能进入正式实现。

| 模式 | 主 PRD 中的职责 | 独立策划文档 |
| --- | --- | --- |
| 自由模式 | v1.4.19 主玩法迁移基准，验证平台壳不回退 | 暂不单独拆分 |
| 每日挑战 | 定义日留存入口、同题公平、日榜数据边界 | `docs/modes/daily-challenge-design-v1.0.md` |
| 盲拼模式 | 定义硬核挑战入口、评级数据边界、与自由模式隔离 | `docs/modes/blind-mode-design-v1.0.md` |
| 禅意模式 | 定义低压力入口、沉浸数据边界、非竞技体验 | `docs/modes/zen-mode-design-v1.0.md` |

模式策划必须遵守一个底线：**新增模式不能只是自由模式的 UI 裁剪版。** 每个模式都必须拥有独立的玩家承诺、核心循环、失败处理和回访理由。

## 8.1 自由模式

### 8.1.1 定位

自由模式是探索、练习和功能展示台。它继承当前主游戏体验，是新用户理解规则的第一落点。

### 8.1.2 规则

- 玩家自由选择难度、切割类型、形状类型。
- 显示全量 HUD：分数、计时、提示、难度系数、排行榜入口。
- 成绩写入自由模式排行榜。
- 记录最近一次配置，用于 Lobby 快速继续。

### 8.1.3 v2.0 改动

- 当前 `/` 直接游戏页迁移到 `/play/free`。
- 自由模式作为默认模式配置，验证 ModeController 可承载现有体验。
- 自由模式排行榜不再与每日挑战或盲拼数据混用。

### 8.1.4 验收标准

- 玩家从 Lobby 点击自由模式可进入完整游戏。
- 当前 v1.4.19 的自由玩法不发生明显功能倒退。
- 自由模式完成后只影响自由模式记录和自由榜。

---

## 8.2 每日挑战

每日挑战在主 PRD 中只确定以下平台边界：

- 路由：`/play/daily`、`/play/daily/result`。
- 数据归属：`game_mode = DAILY`，强制携带 `challenge_date`。
- 公平前提：必须使用 seedable PRNG 和服务端每日参数。
- 入口职责：承担日留存、轻社交和同题竞速。
- 不可同质化要求：每日挑战必须拥有“今日唯一题目、状态机、日榜、重拼代价、回顾页面”，不能只是锁定参数的自由模式。

详细玩法定案见 `docs/modes/daily-challenge-design-v1.0.md`。

---

## 8.3 盲拼模式

盲拼模式在主 PRD 中只确定以下平台边界：

- 路由：`/play/blind`。
- 数据归属：不进入自由模式分数榜，写入盲拼评级档案。
- 入口职责：承担硬核技巧、空间记忆和长期技能天花板。
- UI 原则：隐藏或弱化自由模式的分数刺激，突出时间、提示、评级和降级完成。
- 不可同质化要求：盲拼模式必须改变玩家解题信息结构，不能只是“自由模式隐藏几个按钮”。

详细玩法定案见 `docs/modes/blind-mode-design-v1.0.md`。

---

## 8.4 禅意模式

禅意模式在主 PRD 中只确定以下平台边界：

- 路由：`/play/zen`。
- 数据归属：不进入任何排行榜，只累计禅意时长、环境音和主题解锁。
- 入口职责：承担低压力、沉浸、截图传播和情绪缓冲。
- UI 原则：极简、低干扰、低评价感。
- 不可同质化要求：禅意模式必须改变游戏的节奏、目标和感官反馈，不能只是“自由模式关闭计分器”。

详细玩法定案见 `docs/modes/zen-mode-design-v1.0.md`。

## 8.5 平台扩展位：Reward Activity Templates

### 8.5.1 定位

大厅上线后，Generative Puzzle 不再只有“进入某一局拼图”的入口能力，而具备承载轻量活动的 Activity Hub 条件。

`Reward Activity Templates` 是 v2.0 预留的平台扩展位，用于未来挂载非拼图类轻量奖励玩法。它们不是第五个核心模式，也不替代四个拼图主模式，而是服务于节奏调剂、节日运营、身份表达和奖励发放。

v2.0 首发只预留架构概念和产品边界，不实现具体 Reward Activity。

### 8.5.2 与核心模式的关系

| 类型 | 作用 | 是否属于主模式 | 是否影响核心成绩 |
| --- | --- | --- | --- |
| 自由 / 每日 / 盲拼 / 禅意 | 承载拼图主体验 | 是 | 是，按各自规则影响 |
| Reward Activity | 承载轻量奖励互动 | 否 | 否 |
| 拼图师档案 | 聚合身份和进度 | 元系统 | 间接影响解锁 |

Reward Activity 必须依附于大厅、档案或结算流出现，不应挤占四个主模式的入口权重。

### 8.5.3 可选模板方向

| 模板 | 描述 | 奖励类型 |
| --- | --- | --- |
| 记忆闪卡 | 短时间观察形状后选择正确轮廓 | 称号进度、轻量粒子 |
| 色彩调律 | 在限定色盘中调出指定氛围配色 | 禅意主题、背景色组 |
| 声音收集 | 完成短互动后解锁环境音碎片 | 环境音、音效包 |
| 节日仪式 | 节日限定点击、拖拽或收集互动 | 限定边框、截图贴纸 |
| 奖励房间 | 完成高价值拼图后进入短暂展示空间 | 收藏物、档案徽章 |
| 每日签章 | 完成每日挑战后盖章，不做抽卡赌博化 | 日历视觉、连续记录 |

这些模板可以不是拼图，但必须和“形状、颜色、声音、节奏、空间记忆、拼图身份”保持主题关联。

### 8.5.4 奖励边界

Reward Activity 只允许发放表达型和收藏型奖励：

- 主题配色。
- 环境音。
- 结算粒子。
- 头像边框。
- 档案称号。
- 节日贴纸。
- Lobby 卡片皮肤。
- 收藏物图鉴。

Reward Activity 不允许发放或影响：

- 每日挑战排名优势。
- 盲拼评级条件。
- 自由模式分数倍率。
- 磁吸阈值、提示次数、旋转步长等核心手感参数。
- 任何会造成付费或活动玩家竞技优势的能力。

### 8.5.5 触发入口

Reward Activity 可以出现在以下位置：

| 入口 | 使用场景 |
| --- | --- |
| Lobby 活动位 | 节日活动、限时主题、轻量收集 |
| 每日挑战完成后 | 签章、日历、轻奖励 |
| 禅意模式累计时长后 | 声音收集、主题碎片 |
| 盲拼 S 级后 | 高价值展示房间 |
| `/profile` 档案页 | 收藏物查看和进度回顾 |

首屏 Lobby 的主入口仍优先展示四个拼图模式。活动位只能作为次级入口。

### 8.5.6 技术预留

未来可定义 `ActivityTemplate` 注册表：

```ts
type ActivityTemplate = {
  id: string;
  type: 'memory_card' | 'color_tuning' | 'sound_collect' | 'festival_ritual' | 'reward_room' | 'daily_stamp';
  entry: 'lobby' | 'post_daily' | 'post_blind' | 'zen_milestone' | 'profile';
  availability: {
    startsAt?: string;
    endsAt?: string;
    requiredUnlockKey?: string;
  };
  rewardPolicy: {
    rewardKeys: string[];
    affectsLeaderboard: false;
    affectsBlindGrade: false;
    affectsScoreMultiplier: false;
  };
};
```

Reward Activity 的数据应独立写入活动记录或档案奖励表，不写入拼图主会话表，避免污染模式数据。

### 8.5.7 验收原则

未来任何 Reward Activity 上线前必须满足：

- 不降低四个主模式的入口清晰度。
- 不影响排行榜、评级和核心成绩公平性。
- 玩法时长短，默认 10-60 秒内完成。
- 奖励可展示、可收藏，但不提供竞技优势。
- 可以被关闭、下线或替换，不破坏主游戏流程。

# 9. 拼图师档案

## 9.1 定位

拼图师档案不是 RPG 经验条，而是玩家行为的镜子。它回答：我是怎样的拼图玩家？

## 9.2 档案维度

| 维度 | 来源 | 指标 |
| --- | --- | --- |
| 探索深度 | 自由模式 | 尝试过的切割、形状、最高难度、累计完成 |
| 日常节奏 | 每日挑战 | 连续打卡、首次完成数、重拼数、最高排名 |
| 盲拼技艺 | 盲拼模式 | S 级覆盖率、最高评级、零提示连续完成 |
| 禅意时光 | 禅意模式 | 累计时长、常用环境音、沉浸称号 |

## 9.3 解锁体系

| 条件 | 解锁 | 类型 |
| --- | --- | --- |
| 完成首局自由模式 | 盲拼入门入口 | 引导 |
| 每日挑战连续 7 天 | 周常边框、高级禅意环境音 | 表达 |
| 每日挑战连续 14 天 | 特约拼图师称号、仪式豁免 | 特权 |
| 盲拼首次 S 级 | 金色结算粒子 | 视觉 |
| 禅意累计 1 小时 | 雨声/远雷环境音 | 听觉 |
| 自由模式完成 9 种切割和形状组合 | 探索者称号 | 身份 |
| 盲拼 S 级覆盖率 >= 50% | 极夜禅意主题 | 终局联动 |

## 9.4 展示原则

- Lobby 只展示轻量进度，不弹窗打扰。
- `/profile` 展示完整进度和解锁条件。
- 进度透明，但不制造过强红点压力。

# 10. 数据架构

## 10.1 核心实体

### game_sessions

追加式原始会话记录。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | uuid | 会话 ID |
| user_id | uuid | 用户 ID |
| idempotency_key | text | 幂等键 |
| game_mode | enum | FREE / DAILY / BLIND / ZEN |
| challenge_date | date nullable | DAILY 使用 |
| attempt_no | int nullable | DAILY 使用，1 或 2 |
| attempt_status | text | completed / dnf / downgraded |
| seed | text nullable | 生成种子 |
| seed_version | text nullable | 每日题版本 |
| puzzle_fingerprint | text nullable | 拼图逻辑指纹 |
| difficulty | text nullable | 难度 |
| cut_type | text nullable | 切割类型 |
| shape_type | text nullable | 形状类型 |
| duration_ms | int | 时长 |
| score | int nullable | 分数模式使用 |
| blind_grade | text nullable | 盲拼评级 |
| is_retry | boolean | 每日重拼标识 |
| metadata | jsonb | 兼容扩展 |
| client_created_at | timestamptz | 客户端时间 |
| uploaded_at | timestamptz | 上传时间 |

### public_leaderboard_entries

排行榜投影表，不直接暴露完整会话。

| 字段 | 说明 |
| --- | --- |
| game_mode | FREE 或 DAILY |
| challenge_date | DAILY 必填，FREE 为空 |
| display_name | 展示名 |
| best_score | 最佳分 |
| best_duration_ms | 最佳时长 |
| is_retry | 是否重拼成绩 |
| puzzle_params | 参数摘要 |

### daily_challenge_states

每日挑战状态表。

| 字段 | 说明 |
| --- | --- |
| user_id | 用户 |
| challenge_date | 日期 |
| state | NOT_STARTED / PLAYING / FIRST_WIN / FIRST_DNF / RETRY_WIN / RETRY_DNF |
| first_started_at | 首次开始 |
| first_completed_at | 首次完成 |
| retry_started_at | 重拼开始 |
| retry_completed_at | 重拼完成 |

### blind_profile_records

盲拼档案投影。

| 字段 | 说明 |
| --- | --- |
| user_id | 用户 |
| difficulty | 难度 |
| cut_type | 切割 |
| best_grade | 最佳评级 |
| best_duration_ms | 最佳时长 |
| s_achieved_at | 首次 S 时间 |
| attempts_count | 尝试次数 |

### zen_user_stats

禅意时长聚合。

| 字段 | 说明 |
| --- | --- |
| user_id | 用户 |
| total_duration_ms | 累计时长 |
| last_session_at | 最近一次 |
| unlocked_audio_keys | 已解锁环境音 |
| unlocked_theme_keys | 已解锁主题 |

## 10.2 约束策略

推荐使用模式化唯一索引，而不是单一大唯一键。

```sql
-- DAILY: 每个用户每天最多保留一个公开排行榜投影
create unique index public_leaderboard_daily_user_date_uq
  on public.public_leaderboard_entries (user_id, game_mode, challenge_date)
  where game_mode = 'DAILY';

-- FREE: 每个用户每组参数一个最佳投影
create unique index public_leaderboard_free_user_params_uq
  on public.public_leaderboard_entries (user_id, game_mode, difficulty, cut_type, shape_type)
  where game_mode = 'FREE';

-- BLIND: 每个用户每组盲拼组合一个档案投影
create unique index blind_profile_user_combo_uq
  on public.blind_profile_records (user_id, difficulty, cut_type);
```

# 12. 技术实施路径

## R0 Foundation：平台底座

目标：让项目具备模式化承载能力。

范围：

- 新增 `GameMode` 类型和 `ModeConfig`。
- 新增 Lobby 路由和 `/play/free`。
- 当前主游戏迁移到自由模式。
- 引入 `RandomSource`，完成生成链路第一轮替换。
- Supabase schema 增加模式字段，保持向后兼容迁移。

验收：

- 自由模式体验不回退。
- 同 seed 同参数可生成同一逻辑拼图。
- 自由模式数据写入不污染旧榜。

## R1 Launch：每日挑战 Lite + 禅意模式

目标：上线 v2.0 首批用户可感知价值。

范围：

- Lobby 四模式入口。
- 每日挑战 Lite。
- 禅意模式基础版。
- 每日结果页。
- 基础模式隔离结算。

验收：

- 玩家每天可完成同题挑战。
- 已完成每日挑战后路由守卫生效。
- 禅意模式无分数、无榜单、时长可累计。

## R2 Expansion：盲拼模式完整 MVP

目标：建立项目技能天花板。

范围：

- 3 秒记忆窗口。
- 隐藏轮廓。
- 90 度旋转。
- 缩小磁吸阈值。
- 提示和评级。
- 降级完成。

验收：

- 盲拼结果不写入自由榜，除非走降级完成策略。
- S/A/B/C/D 判定稳定。
- 盲拼档案可统计覆盖率。

## R3 Profile：拼图师档案

目标：把行为转化为长期身份表达。

范围：

- `/profile` 页面。
- 四维档案指标。
- 解锁进度。
- 基础称号和视觉奖励。

验收：

- 玩家能看到四模式进度。
- 解锁条件透明。
- Lobby 可读取轻量档案摘要。

## R4 Operations：自适应和运营

目标：从功能平台进入运营平台。

范围：

- 每日挑战 7 日动态难度。
- 人工覆盖每日题。
- Rage Quit 关怀提示。
- 模式后台深度分析。
- 分享图和历史挑战回放。

验收：

- 运营能按日期查看挑战数据。
- 动态难度调整有日志和回滚。
- 玩家侧无突然暗改体验。

# 13. 风险与对策

| 风险 | 等级 | 对策 |
| --- | --- | --- |
| PRNG 重构影响核心玩法 | 高 | 先加兼容接口，逐模块替换；用同 seed 快照测试保护 |
| P0 范围过大 | 高 | 拆为 R0 和 R1，R0 不追求新模式完整上线 |
| 模式壳过度抽象 | 中 | 只抽象真实差异：UI、规则、结算、持久化 |
| 新模式策划不足导致同质化 | 高 | 每日、盲拼、禅意必须先通过独立策划文档评审，再进入实现 |
| 禅意模式过弱 | 中 | 用音效、粒子、截图完成反馈提升传播性 |
| 每日挑战作弊 | 高 | 服务端参数、fingerprint、时长区间、attempt 状态共同校验 |
| 新手被模式大厅吓退 | 中 | Lobby 默认突出“继续探索”和“放松一下” |
| 盲拼难度过高 | 中 | 入门引导、免费轮廓闪现、降级完成保护 |

# 15. 推荐首发切片

v2.0 首发建议不追求四模式全部完美，而是发布“平台骨架 + 两个强感知模式”。

每日挑战和禅意模式是否进入首发，以对应独立策划文档完成评审后的 MVP 范围为准。若玩法验证不足，首发应只发布平台骨架、自由模式迁移和 PRNG 数据底座。

## 首发包含

1. Lobby。
2. `/play/free` 自由模式迁移。
3. PRNG 确定性生成底座。
4. 每日挑战 Lite。
5. 禅意模式基础版。
6. 基础数据隔离。

## 首发不包含

1. 完整盲拼评级。
2. 复杂档案页。
3. 动态难度算法。
4. 盲拼挑战日。
5. 完整运营覆盖接口。

这样可以让玩家立刻感知 v2.0 的变化，同时把最高风险的算法和数据边界先打稳。
