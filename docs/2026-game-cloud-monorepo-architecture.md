# game-cloud 单仓多包架构方案（2026）

> **状态**：已采纳（方案 1：单仓多包）  
> **关联分支**：`game-only`（纯游戏交付）→ `game-cloud`（云端进阶）  
> **文档版本**：v1.1  
> **目的**：把「瘦身后的纯游戏核心」与「云端基础设施」在工程结构上分离，并固定两项关键架构决策：**同步权威顺序**、**删除/合规（RPC 为主）**。

---

## 0. 已确认决策（2026-04-03）

以下经产品/技术确认，作为实现与 Code Review 的基准；变更需走文档版本与 `CHANGELOG`。

| 决策项 | 结论 |
|--------|------|
| **同步权威顺序** | 采用 **§3** 全文：单局与服务端聚合以服务端为准，本地为缓存；合并顺序与冲突策略以 **§3.2、§3.3** 为准。 |
| **删除「本应用游戏数据」** | 采用 **Postgres RPC（`security definer`）**，与 RLS 一致；见 **§4**。 |
| **账号注销 / Auth 删号 / 导出邮件等** | **暂不实现**；需求优先级 **极低**。若未来需要，再补 **Supabase Edge Functions**（或受控服务端 + service role），见 **§4.5**。 |

## 1. 背景与目标

- **game-only**：可独立运行、可发布的纯游戏，依赖面最小。  
- **game-cloud**：在保留同一套游戏核心的前提下，增加账户、多端数据一致、公开榜等能力。  
- **问题**：若继续在单一 `app` 内「贴片式」接 Supabase，领域逻辑与基础设施会持续纠缠（参见 `docs/2026-game-cloud-architecture-review.md`）。

**本方案目标**：

1. 采用 **单仓多包（monorepo packages）**，用依赖方向约束替代口头约定。  
2. 书面固定 **同步权威顺序**，避免产品与实现各说各话。  
3. 书面固定 **删除/合规** 的技术选型（RPC vs Edge Functions），并说明何时用哪一种。

---

## 2. 单仓多包总览（已采纳）

### 2.1 推荐目录结构（目标态）

以下为 **目标布局**；迁移可渐进，不必一次性搬完。

```text
packages/
  game-core/          # 领域：规则、计分、状态机类型、纯函数（禁止 React / Supabase / window）
  game-ui/            # 可复用 UI 与 Canvas 组件（可依赖 game-core）
apps/
  web/                # 当前 Next 应用：装配路由、Auth、Supabase 实现（或拆为 web-game-only / web-cloud 视需要）
```

若短期内仍只有一个 Next 应用，可先在仓库根用 **`packages/game-core` + `packages/game-ui`**，应用留在 `app/`，通过 workspace 引用包；待边界稳定再考虑 `apps/web-cloud` 与 `apps/web-game-only` 双应用。

### 2.2 依赖规则（必须可执行）

| 包 | 允许依赖 | 禁止 |
|----|-----------|------|
| `game-core` | 仅 TS 标准库、与平台无关的工具 | `react`、`next`、`@supabase/*`、`localStorage` |
| `game-ui` | `game-core`、`react` | `@supabase/*`、直接访问 `localStorage`（除极窄封装） |
| Next `app` / 装配层 | 上述包、`@supabase`、同步适配器 | 在页面里写大段计分/切割算法 |

**落地建议**：在 ESLint 中对 `packages/game-core` 配置 `no-restricted-imports`，禁止 `@supabase`、`next/*`、`react`。

### 2.3 端口（Ports）位置

- **接口定义**建议放在 `game-core` 或独立薄包 `game-sync-contracts`（二选一，避免循环依赖）。  
- **Supabase 实现**放在应用侧或 `packages/infra-supabase`，仅实现端口，不反向污染 `game-core`。

## 3. 架构决策一：同步权威顺序（已确认）

本节回答：**谁说了算、按什么顺序合并**，用于指导 `GameDataManager`、离线队列与 UI 表现。

### 3.1 数据分类与权威来源

| 数据类型 | 权威来源 | 说明 |
|----------|----------|------|
| **单局原始记录**（一次 `game_sessions` 插入） | **服务端（Postgres）** | 以库内唯一约束 + `idempotency_key` 判重；客户端只「提议」，不假定一定写入成功。 |
| **聚合公开榜**（`public_leaderboard_entries` / `leaderboards` 视图） | **服务端（触发器/函数维护）** | 客户端 **只读**；不在本地伪造「全球排名」为真值。 |
| **本地 `leaderboard` / `history`（localStorage）** | **本地缓存 + 展示用** | 用于离线体验与首屏速度；登录后与云端 **按规则合并**，不是全局真理。 |
| **离线队列** | **客户端暂存** | 网络恢复后上传；成功后以服务端存在性为准（重复键视为已成功同步）。 |

### 3.2 同步流程（顺序，已确认）

以下顺序兼顾体验与一致性，并与当前实现方向兼容（实现可逐步对齐本文）。

1. **局内结束（立刻）**  
   - 先写本地（保证单机与重进页面仍有记录）。  
   - 再异步上传会话（失败则进入 **离线队列**，不阻塞游戏流程）。

2. **网络从离线 → 在线**  
   - 先 **drain 离线队列**（仍带同一套 idempotency，避免重复插）。  
   - 再按需 **拉取个人历史**（若已登录）。

3. **用户登录成功瞬间**  
   - 先 **同步离线队列**（把未上传的本地局交给服务端）。  
   - 再 **拉取该用户 `game_sessions` 衍生记录**（或等价 API）。  
   - 再执行 **本地合并**（见 3.3）。  
   - **一次性迁移**（本地历史 → 云）若保留：仅在「未标记完成且需要」时跑；**迁移完成标记**应在 **关键条件满足** 后写入（例如队列清空且拉取成功，或分阶段重试策略），避免「全失败仍标完成」。

4. **展示层**  
   - **我的历史/个人榜**：合并后以 **服务端存在 + 本地未上传** 的并集展示，重复键以服务端为准。  
   - **全球公开榜**：仅展示 **服务端视图**，不做「本地覆盖服务端」。

### 3.3 合并规则（简版）

- **同一逻辑局**：以 `idempotency_key` 或服务端 `id` 去重；时间戳接近且分数相同可作为辅助去重（与现实现一致时可保留）。  
- **冲突策略**：同一键 **服务端 wins**；本地仅保留「尚未获得服务端 id / 未确认写入」的 pending 行（若未来引入显式状态字段则更清晰）。

### 3.4 与「纯游戏」分支的关系

- **game-only**：不启用上传；端口实现为 **no-op**，本地仍为权威（仅本地）。  
- **game-cloud**：启用上述顺序；**不改变** `game-core` 内规则与计分，只改变 **装配与同步适配器**。

## 4. 架构决策二：删除/合规 —— RPC 与 Edge 的分工（已确认）

### 4.1 结论与实施节奏（已确认）

| 场景 | **已定方案** | **实施节奏** | **要点** |
|------|----------------|--------------|----------|
| **删除本应用游戏数据**（`game_sessions`、本人相关聚合行等，**不删 Auth 账号**） | **Postgres RPC（`security definer`）+ RLS** | **当前主线** | 单事务、低延迟、易审计；函数内用 `auth.uid()` 强校验。 |
| **账号级注销 / 导出 / 外部合规** | **Edge 或受控服务端 + service role** | **暂缓**（需求极低） | 需 Admin API 或外部系统时再实现；见 §4.5。 |

**原则**：库内可事务化删除走 **RPC**；需 **提权或外部世界** 时走 **Edge/服务端**——暂缓项仅影响排期，不改变技术选型方向。

### 4.2 为何不全用 Edge

- Edge 适合编排，但 **多表一致删除** 若在 Edge 里多次往返，需自行保证事务语义；而 **RPC 内 `SECURITY DEFINER` + 单事务** 更直接。  
- Edge 持有 **service role** 的密钥与部署面更大，应 **少暴露端点、每个端点做鉴权与限流**。

### 4.3 为何不全用 RPC

- **删除 `auth.users` 用户** 通常需 **Supabase Admin**，客户端不能完成；用 RPC 从浏览器直接删也不安全。  
- **跨服务合规**（邮件、审计日志到外部 SIEM）更适合 **Edge / 后端服务**。

### 4.4 实施要点（与现有 SQL 一致）

- 当前 MVP SQL 倾向 **append-only `game_sessions`**：若产品需要「用户清空自己的云端游戏数据」，应 **显式增加**「用户可删本人数据」的策略或 **封装在 RPC** 中执行删除，并与 RLS 一致，避免出现「UI 有按钮但永远 403」的情况。  
- 聚合表由触发器维护时，删除 `game_sessions` 后应 **刷新或级联** 聚合逻辑（已有 `refresh_leaderboard_for_user` 思路可复用）。

### 4.5 账号注销与 Edge（暂缓）

- **现状**：不实现「一键注销账号」、不向 Auth 侧删用户。  
- **触发条件**：产品明确要求 **GDPR 式删号**、或必须调用 **Admin API** 时，再开独立任务：部署 **Edge Function**（或等价受控服务端），密钥仅存服务端，并补审计与限流。  
- **与 §0 对齐**：在此之前，**§4.1 表格中「账号级」一行保持为「暂缓」**。

## 5. 整合落地路线（与分支策略对齐）

### 阶段 A：工程骨架

- 建立 `packages/game-core`（先从 **类型 + 计分 + 与云无关的纯函数** 迁入）。  
- 配置 workspace（npm workspaces / pnpm workspace 等，与仓库现有包管理一致）。  
- 应用内引入 **端口接口**，`CloudGameRepository` 变为 **端口的 Supabase 实现**。

### 阶段 B：Context 减负

- 保证 **游戏 Reducer** 与 **同步副作用** 分离（`useCloudSync` 一类钩子保持在外层）。  
- 清理评审中已记录的 **P0 游戏逻辑缺陷**（如重复 `case`），避免架构迁移时放大 bug。

### 阶段 C：服务端对齐

- 按 §4 实现 **删除游戏数据**：以 **RPC** 为唯一客户端调用路径（与 §0、§4.5 一致）。  
- **账号注销 / Edge**：不纳入本阶段；见 §4.5。  
- RLS、策略与文档 **同学变更**，并在 CI 或发布检查清单中增加「策略与客户端一致」项。

### 阶段 D：质量闸门（轻量）

- 不追求恢复历史全部测试矩阵；对 **合并规则、幂等键、RPC 权限** 做 **少量单测/集成测** 即可。

## 7. 认证与身份：匿名身份 + 昵称/PIN 恢复设计（草案）

> **目标**：满足小游戏的“无感可玩”，同时提供“可找回”的最低成本方案，且不破坏 §3 的同步权威顺序与 §4 的安全边界。

### 7.1 设计原则

- **无感优先**：用户打开网页即可玩；不要求提前登录。  
- **云端主键统一**：一旦启用云端存储，**身份主键以 Supabase `auth.users.id`（UUID）为准**，避免“本地 UUID 当账号”导致的不可控安全与冲突。  
- **恢复是“找回云存档”，不是“证明你是谁就能任意冒用”**：昵称 + PIN 只能作为低门槛恢复因子，应配合风控/限流，并明确产品边界。  
- **与同步一致性兼容**：匿名身份仍遵守 §3：单局与聚合以服务端为准、本地仅缓存；离线队列在有身份后可补传。

### 7.2 身份模型（推荐）

#### A. 云端身份（权威）

- **`supabase.auth` 用户**：每个玩家对应一个 `user.id`（UUID）。  
- **写入关联**：`game_sessions.user_id` 外键指向 `auth.users(id)`。

#### B. 本地身份（辅助，不作账号）

- **`localDeviceId`（UUID）**：仅用于设备级统计、调试、灰度与 UX 体验（例如提示“本设备已设置恢复信息”）。  
- 约束：`localDeviceId` **不能**作为云端数据权限判断依据。

### 7.3 “匿名身份”（无感登录）方案

**推荐做法**：使用 Supabase 的匿名用户能力（匿名会话持久化在浏览器）。当用户首次进入游戏、且本地无会话时，自动创建匿名会话：  

- **用户体验**：无弹窗、无输入；同浏览器后续访问自动识别同一用户。  
- **数据收益**：你能从第一局开始就把记录写入云端（若网络允许），离线则进队列，恢复网络后补传。  

> 备注：若后续需要“升级账号”（绑定 Google/Apple/GitHub），可在不丢存档的前提下，把匿名身份升级为正式身份（具体实现依赖 Supabase 的账号链接能力与产品策略）。

### 7.4 “昵称 + 4 位 PIN”恢复（跨设备找回云存档）

#### A. 建议的数据结构

在 `profiles`（或单独表 `recovery_profiles`）中增加字段（示例）：  

- `user_id uuid primary key references auth.users(id)`  
- `display_name text not null`  
- `pin_hash text not null`（**不存明文 PIN**；使用强 hash，如 bcrypt/argon2，或至少 `sha256(salt + pin)`）  
- `pin_salt text not null`（若使用 sha256 方案）  
- `updated_at timestamptz`  
- 可选冗余展示：`best_score int`、`sessions_count int`（用于“重名靠分数区分”的候选列表展示；也可运行时聚合）

RLS 建议：  

- `profiles` 仍保持 “本人可读写本人行”。  
- “恢复查询”不要直接开放 `select where display_name = ...` 给匿名客户端，否则会变成撞库入口；应走 **RPC**（见下）。

#### B. 恢复流程（推荐：RPC + 候选选择）

1. **用户在新设备点击“找回存档”**，输入：昵称 + 4 位 PIN。  
2. 客户端调用 **RPC**：`recovery_lookup(display_name, pin)`：  
   - 服务端验证 PIN（hash 比对）后，返回 **候选列表**：`user_id` +（最佳分数/局数/更新时间/难度分布摘要）。  
   - 若“昵称+PIN”匹配多条（可允许），客户端展示让用户选择一条（“重名靠分数区分”）。  
3. 用户选中某个候选后，再调用 RPC：`recovery_claim(target_user_id)`：  
   - **产品策略 1（MVP：导入/合并，不切换身份）**：不做“切换登录为 target_user”，而是 **拉取该 user 的历史并合并到当前会话**（作为一次性迁移/导入）。  
     - 优点：实现简单、安全风险小，不需要任何提权能力。  
     - 缺点：后续上传仍归属“当前设备的会话 user_id”，跨设备想同步需要再次导入。  
   - **产品策略 2（完整形态：切换身份，后续上传都归 target UUID）**：找回成功后，让当前设备 **真正登录成 `target_user_id`**，使后续 `game_sessions.user_id = auth.uid()` 恒成立。  
     - 关键约束：在 Supabase/RLS 模型下，**无法仅靠客户端或普通 RPC 直接“变更为另一个 user_id”**；必须由受控服务端能力发放一次性会话（或走第三方 OAuth 绑定/登录）。  
     - 推荐实现：`recovery_claim` 仅做鉴权与风控，随后由 **受控服务端**（Edge Function 或等价服务端路由）为 `target_user_id` 生成一次性登录凭证/会话，再由客户端完成会话切换。  
     - 风险控制：必须配合更严格的限流、审计、失败重试与防撞库策略（见 7.4C）。

> 为什么推荐“导入/合并”而不是“直接变更登录身份”：昵称 + 4 位 PIN 的强度太弱，直接等价“账号密码”会带来明显的冒用风险与客服成本。

#### B.1 推荐的产品策略组合（与你的目标对齐）

为了实现你描述的目标：**“找回后即关联云端 UUID，并持续把本地数据合并后上传到该 UUID，使不同设备/浏览器都能同步并参与全球榜”**，推荐采用：

- **默认无感**：匿名身份自动建立（§7.3）。  
- **找回入口（低门槛）**：昵称 + PIN 用于定位目标存档（§7.4B）。  
- **真正跨设备持续同步（强关联）**：在找回流程中增加“升级/绑定”步骤：  
  - **优先**：绑定第三方 OAuth（Google/Apple/GitHub）作为强身份；一旦绑定，跨设备只需 OAuth 登录即可恢复同一 `user_id`。  
  - **其次（若坚持纯 PIN 找回）**：需要受控服务端发放会话（即 §7.4B 的策略 2）。该能力不属于“纯 RPC”范畴，应单独立项并严格风控。

#### C. 风控与限流（必须）

- RPC 侧对 `recovery_lookup` 增加 **IP/设备** 维度限流（例如 5 次/小时）。  
- 对错误返回保持模糊（例如统一“信息不匹配”），避免泄露“昵称是否存在”。  
- 记录审计日志（最少记录 `timestamp`, `client_ip`, `display_name`, `result`）。

### 7.5 与产品体验的对应关系

- **无感可玩**：匿名会话自动建立，不阻塞游戏流程。  
- **完成后引导登录/设置恢复信息**：仅在游戏完成或点击“全球榜”时提示：  
  - 设置昵称+PIN（用于跨设备恢复与参与公开榜展示名）；或  
  - 绑定第三方 OAuth（用于更强的账号恢复能力，后续可选）。  
- **登录后回到当前页面**：OAuth 登录（若启用）统一走 `redirectTo=<absolute callback>?next=<当前路径>` 的回跳策略；不以 `/scores` 作为公开入口。
