## 0. 执行摘要（给决策层）

本项目已具备较强的功能完成度，但在**架构边界、状态管理、发布安全、工程一致性**上存在结构性风险。  
最大问题不是“某个函数写得不好”，而是**核心状态与副作用耦合过深**，导致：

- 运行正确性风险：Reducer 内副作用、重复 case、复杂状态分支。
- 性能风险：单体 Context 触发高频全树重渲染。
- 安全风险：内部测试页面和性能数据进入公开静态产物。
- 演进风险：`core/`、`utils/`、`hooks/` 之间边界模糊，后续改动成本越来越高。

**结论**：建议按“先止血、再解耦、后治理”的阶段化路线执行；不按自然时间分配任务，而按节点门禁推进，满足验收即进入下一节点。

---

## 1. 评审方法与2026基准

本次方案采用以下基准：

1. **React 19 并发语义**：Reducer 必须纯函数；副作用必须移入 Effect/Action 层。
2. **Next.js 15 生产实践**：公开构建产物最小化；内部工具不得随静态站点暴露。
3. **前端分层架构**：UI 宿主层 / 应用编排层 / 领域内核（纯 TS）边界清晰。
4. **状态治理**：高频状态与低频状态分治（Context + Store 切片化）。
5. **安全默认开启**：StrictMode、最小暴露、日志与观测分级、CI/Lint/Type 强约束。

## 2. 当前代码库客观诊断（基于现状代码）

### 2.1 P0（必须立即处理）

#### P0-1 发布面暴露内部能力
- `app/test/page.tsx` 是完整性能仪表板（依赖 `recharts`），当前存在于主路由树。
- `scripts/build-for-github.cjs` 仅处理 `app/api`，未处理 `app/test`。
- `public/performance-data.json` 作为公开静态文件可被直接访问。

**影响**：内部质量能力、运行特征数据、环境细节被公开，增加攻击面与信息泄露面。

#### P0-2 Reducer 内副作用与动态依赖
在 `contexts/GameContext.tsx` 的 `gameReducer` 内存在：
- `GameDataManager.getLeaderboard()` / `saveGameRecord()` 调用；
- `require("@/utils/score/ScoreCalculator")` 动态导入；
- 多处 `console.log` 调试输出。

**影响**：并发渲染语义下可能重复执行副作用，导致记录重复、状态竞争、行为不可预测。

#### P0-3 `RESTART_GAME` 重复分支
`GameContext.tsx` 内 `RESTART_GAME` 出现两次，后者无法触达。

**影响**：重开流程可能与预期不一致，且未来维护极易误判。

---

### 2.2 P1（高优先级架构债）

#### P1-1 单体 Context 过载
`contexts/GameContext.tsx` 近 2000 行，混合：
- 拼图数据、计时统计、排行榜、音效触发、适配逻辑、持久化。

**影响**：任何状态更新都放大重渲染范围，调试和测试难度指数增长。

#### P1-2 交互 Hook 过于臃肿
`hooks/usePuzzleInteractions.ts` 体量巨大，且存在双数据来源（props + `useGame()`）。

**影响**：状态来源不单一，交互缺陷难定位，复用困难。

#### P1-3 事件系统可回收性不足
`core/EventManager.ts` 中有定时任务与全局监听组织复杂，清理策略不完整。

**影响**：页面生命周期切换与长时运行后，存在性能抖动和内存风险。

---

### 2.3 P2（工程一致性问题）

- `next.config.mjs` 当前 `reactStrictMode: false`。
- `tsconfig.json` 中 `target: "ES6"` 与现代运行环境不匹配，且测试类型混入主编译域。
- CI Node 版本不统一（`deploy.yml` 为 18，`quality-check.yml` 为 20）。
- `README.md` 与实际工程策略存在偏差（如环境基线、流程口径）。

## 3. 目标架构（整改后）

### 3.1 分层目标

### A. 宿主层（Host/UI）
- 目录：`app/`, `components/`, `hooks/ui-*`
- 职责：渲染、交互、输入输出、平台 API 适配。
- 约束：不直接承载复杂领域规则。

### B. 应用层（Application）
- 目录：`contexts/`（薄编排） + `services/`（可新增）
- 职责：流程编排、状态汇聚、调用领域能力与持久化。
- 约束：不包含 Canvas 绘制细节和复杂几何算法。

### C. 领域层（Domain Engine）
- 目录建议：`game-engine/`（可逐步从 `utils/puzzle|shape|geometry|score|difficulty` 迁入）
- 职责：纯 TS 规则、无 React/Next 依赖、可独立单测。

### D. 基础设施层（Infra）
- 目录：`core/`, `utils/data/`
- 职责：事件总线、设备能力、持久化、日志。
- 约束：通过接口被应用层调用，不反向依赖领域层。

## 4. 整改路线图（可执行）

## 阶段 0：止血（按节点推进）

1. **关闭公开暴露**
   - 构建脚本中同时排除 `app/test` 与性能数据产物。
   - 若短期无法迁移，至少通过构建开关确保生产不产出。
2. **修复 Reducer 纯度**
   - 将存储读写、排行榜计算触发迁移到 `useEffect`/应用服务。
   - 移除 Reducer 内动态 `require`。
3. **修复重复分支**
   - 合并 `RESTART_GAME` 为唯一入口。
4. **开启守护基线（预备）**
   - 新增“禁止 Reducer 调副作用模块”的 lint 规则（或代码扫描脚本）。

**阶段验收**：
- 公开产物不再包含 `/test` 与 `performance-data.json`。
- `GameContext` Reducer 中零持久化调用。
- `RESTART_GAME` 单分支，回归通过。

---

## 阶段 1：状态与交互解耦（按节点推进）

1. **Context 拆分**
   - `GameBoardContext`：拼图实体与位置。
   - `GameSessionContext`：计时、分数、局状态。
   - `GameUIContext`：弹层、提示、显示控制。
2. **高频状态下沉到切片 Store**
   - 例如：拖拽中间态、hover/selection、实时分数。
   - 目标：避免全局 Context 订阅抖动。
3. **拆分交互 Hook**
   - `usePieceSelection`, `usePieceDrag`, `usePieceRotation`。
   - 统一单一状态来源（禁止 props + context 混用同一语义状态）。

**阶段验收**：
- 主 Context 文件 < 300 行（目标 < 200 行）。
- 拖拽操作帧率明显稳定（实测 FPS 波动下降）。

---

## 阶段 2：领域边界固化（按节点推进）

1. **建立 `game-engine/` 目录**（渐进迁移，不一次性重写）。
2. **设置依赖方向约束**
   - `game-engine` 禁止 import React/Next/DOM/window。  
   - `game-engine` 禁止直接访问 `GameDataManager`。
3. **明确 `core/` 与 `utils/` 职责**
   - `core`：平台运行时能力；
   - `game-engine`：纯算法规则；
   - `utils/data`：持久化与序列化。

**阶段验收**：
- `game-engine` 可独立单测运行。
- 领域规则变更不需要触碰 UI 组件。

---

## 阶段 3：安全与工程治理（按节点推进）

1. **恢复 `reactStrictMode: true`**（前提：阶段 0/1 完成）。
2. **TypeScript 与 CI 基线统一**
   - `tsconfig` 目标升级（建议 `ES2022`）。
   - 测试与构建类型域分离（主编译不混入 E2E 类型）。
   - CI 统一 Node 22 LTS。
3. **生产日志治理**
   - 禁止随意 `console.log`；保留 `warn/error` 分级日志。
4. **构建最小暴露策略**
   - 对外产物仅保留游戏核心能力和必要页面。

**阶段验收**：
- StrictMode 正常开启。
- 主分支 CI（lint/type/test/build）稳定。

---

## 5. 安全整改专项

1. **路由暴露治理**：内部页面不进入公开构建产物。
2. **数据暴露治理**：性能/调试 JSON 不在 `public/` 长期驻留。
3. **持久化输入校验**：读写 `localStorage` 前后统一 schema 校验（建议 Zod）。
4. **错误处理策略**：用户可见错误信息去敏，内部细节仅记录在开发日志。

---

## 6. 性能整改专项

1. **渲染层**：高频状态切片化，避免无关组件重渲染。
2. **事件层**：统一事件订阅生命周期，确保可回收。
3. **算法层**：拼图生成与评分计算做缓存/增量计算评估。
4. **观测层**：保留内部性能采样，但迁移到私有工具链。

---

## 7. 质量门禁（Definition of Done）

每阶段完成必须满足以下门禁：

- 类型：`npm run lint` 与类型检查通过。
- 测试：核心单测 + 关键 E2E（smoke）通过。
- 安全：公开构建产物扫描无内部页面与内部数据。
- 文档：架构图、边界规则、迁移清单同步更新。

---

## 8. 量化目标（KPI）

- `GameContext.tsx`：约 1900 行 → < 300 行（目标 < 200）。
- Reducer 副作用调用：>0 → 0。
- 公开内部页面：存在 → 0。
- 交互关键路径（拖拽）平均 FPS：提升并稳定在目标区间。
- 构建一致性：CI Node 版本统一，主分支无“环境差异型”失败。

## 9. 风险与回滚

### 主要风险
- 大文件拆分引起隐性行为变化；
- 状态迁移期间短期回归问题增加；
- 构建策略调整影响发布节奏。

### 回滚策略
- 每阶段以 feature flag 或独立分支合并；
- 保留旧实现 1~2 个迭代周期（只读模式），出现重大问题可快速切回；
- 生产发布采用小步快跑（每次只引入一个架构变化面）。
